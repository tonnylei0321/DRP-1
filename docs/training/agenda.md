### 10:00 - 10:50 | 理论1：OpenSpec 规范驱动开发（50min）

**目标**：建立"先规范后实现"的心智模型。

**流程**：
1. **10:00-10:15**（15min）— 问题引入
   - 展示一个真实案例：需求理解偏差导致返工
   - "边想边写"的代价：沟通成本、返工成本、知识流失
   - 引入 OpenSpec：让 AI 和人都能读懂的需求规范

2. **10:15-10:35**（20min）— OpenSpec 核心概念
   - **目录模型**：specs/（当前真相） vs changes/（变更历史）
   - **三阶段工作流**：proposal → implementation → archive
   - **文件格式**：
     - proposal.md — 为什么做、做什么、影响什么
     - tasks.md — bite-sized 实现步骤
     - spec.md delta — ADDED / MODIFIED / REMOVED
   - 现场展示一个真实的 proposal.md 示例
   - 现场展示一个真实的 tasks.md 示例

3. **10:35-10:50**（15min）— 互动环节
   - 给出一段模糊需求："优化用户搜索体验"
   - 全员 2 分钟思考：这个需求缺什么？
   - 随机抽 3-4 人回答
   - 讲师展示如何将模糊需求转化为 OpenSpec proposal
   - **关键点**："拆解任务比写代码更重要"

**讲师准备**：
- 真实 proposal.md 和 tasks.md 示例（从项目中选取）
- "模糊需求"到"清晰 proposal"的对比幻灯片

### 10:50 - 11:40 | 理论2：Claude Code + MCP 多AI协同（50min）

**目标**：理解三个 AI 的角色分工和协作模式。

**流程**：
1. **10:50-11:05**（15min）— Claude Code 基础
   - Claude Code 是什么：CLI 交互式 AI 编程助手
   - 核心能力：读/写/搜索代码、运行命令、多工具协同
   - Skill 系统：内置技能库，快捷调用
   - 关键操作演示：
     ```
     # 读文件
     # 搜索代码
     # 编辑文件
     # 运行测试
     ```

2. **11:05-11:20**（15min）— MCP 多AI协同
   - 什么是 MCP（Model Context Protocol）
   - 三个 AI 的角色分工：
     ```
     Claude Code — 主体思考者、决策者、后端主力
     Codex MCP  — 后端技术顾问、交叉检查
     Gemini MCP — 前端主力、大规模分析、全局视角
     ```
   - 协作模式：
     ```
     后端：Claude 实现 → Codex 审查 → Claude 修复
     前端：Claude 设计 → Gemini 实现 → Claude 审查
     分析：Claude 分析 → Codex + Gemini 交叉验证 → Claude 决策
     ```
   - **核心原则**："你是主人，AI 是顾问。先思考，再验证。"

3. **11:20-11:35**（15min）— 关键技巧与刹车机制
   - Prompt 工程要点：给上下文、给约束、给示例
   - **刹车机制**（极重要）：
     - AI 连续 2 次修复失败 → 人类介入
     - AI 生成的代码偏离规范 → 回到 OpenSpec 对齐
     - AI 陷入循环 → Ctrl+C 中断，重新给思路
   - 常见反模式：
     - ❌ "帮我写个登录" — 太模糊
     - ✅ "根据 proposal.md 第 2.1 节，实现用户认证接口" — 有上下文

4. **11:35-11:40**（5min）— Q&A

**讲师准备**：
- MCP 架构图
- 角色分工对照表（打印发放）
- 刹车机制流程图

## 第二部分：分轨实战（13:30 - 17:00）

> **核心流程对齐 CLAUDE.md 规范**：设计先行 → 三方审核 → 提案 → TDD 实现 → 三方代码审查 → 验证完成

### 13:30 - 14:00 | 破冰 + 需求设计（30min）

**目标**：环境就绪、创建分支、完成需求分析和架构设计。

**流程**：
1. **13:30-13:40**（10min）— 题目讲解 + 破冰
   - 讲师讲解题目（见各技术栈 `exercises/{tech}/README.md`）
   - 学员按"破冰第一步"操作：安装依赖、运行测试、启动 Claude Code
   - **关键：每人从 develop 创建自己的特性分支**
     ```bash
     git checkout develop
     git checkout -b feature/training-{你的名字}
     ```

2. **13:40-14:00**（20min）— 需求分析与架构设计（brainstorming）
   - 在 Claude Code 中发送需求，让 Claude 进行需求分析
   - Claude 生成需求理解和组件/模块设计方案
   - **三方设计审核**：让 Claude 调用 Codex 审查技术可行性，调用 Gemini 审查场景覆盖度
   - 三方达成一致后，确认设计方案
   - 产出：设计思路文档（可简化，记在 proposal 的 Why 和 What Changes 中）

**注意**：这一步禁止写业务代码！只做分析和设计。

---

### 14:00 - 14:20 | 编写 OpenSpec 提案（20min）

**目标**：基于设计方案，创建规范的 OpenSpec 提案。

**流程**：
1. **14:00-14:15**（15min）— 编写 proposal.md + tasks.md
   - 基于上一步的设计方案，用 Claude Code 生成 OpenSpec 提案
   - proposal.md：Why（为什么做）、What Changes（做什么）、Impact（影响范围）
   - tasks.md：bite-sized 实现步骤（每步含文件路径、代码要点、验证命令）
   - **三方提案审核**：Codex 审查 API 设计和边界条件，Gemini 审查场景覆盖完整性
   - 三方一致后，提案定稿

2. **14:15-14:20**（5min）— 讲师巡场审核
   - 讲师快速过目每人的 proposal.md 和 tasks.md
   - 常见问题：proposal 太模糊 → 引导细化；tasks 粒度太大 → 帮助拆分
   - **关卡1打卡**：proposal.md + tasks.md 完成，群里发截图

---

### 14:20 - 15:00 | TDD 实现主需求（40min）

**目标**：基于 OpenSpec 提案，按 TDD 流程实现主需求。

**流程**：
1. 按 tasks.md 步骤顺序实现
2. **严格 TDD**：每个 task 先写测试（RED）→ 实现代码（GREEN）→ 重构（REFACTOR）
3. Claude Code 是主要编码工具，但**开发者必须审查每一段生成的代码**
4. 讲师巡场处理卡壳问题
5. **关卡2打卡**：主功能测试通过，群里发截图

---

### 15:00 - 15:15 | 需求变更投放（15min）

**目标**：体验 OpenSpec 的变更管理能力。

**流程**：
1. 讲师宣布需求变更（每组不同的变更内容，见下方表格）
2. **学员更新 proposal.md**：新增变更内容到 What Changes
3. **学员更新 tasks.md**：新增实现步骤
4. **三方变更审核**：Codex/Gemini 审核变更方案的合理性

---

### 15:15 - 16:00 | 变更实现 + 边界处理（45min）

**流程**：
1. 按更新后的 tasks.md 继续 TDD 实现
2. 处理边界条件
3. 补充测试用例

---

### 16:00 - 16:30 | 三方代码审查 + 验证（30min）

**目标**：验证实现符合需求和设计，代码质量达标。

**流程**：
1. **16:00-16:20**（20min）— 三方代码审查
   - **Codex 审查**：代码质量、安全隐患、性能问题、边界条件
   - **Gemini 审查**：功能覆盖度、实现是否偏离 proposal 需求
   - **Claude 综合判断**：根据两方审查意见修复问题
   - 重点检查：实现是否符合 proposal.md 中定义的需求？是否有遗漏的功能点？

2. **16:20-16:30**（10min）— 最终验证
   - 运行全部测试，确认通过
   - 检查 tasks.md 中所有任务已完成（标记 `[x]`）
   - 提交代码到自己的特性分支
     ```bash
     git add .
     git commit -m "feat: 完成训练任务 - {你的题目名}"
     ```
   - **关卡3打卡**：三方审查通过 + 测试全绿，群里发截图

**讲师巡场检查点**：
| 时间 | 检查内容 | 发现问题怎么办 |
|------|---------|--------------|
| 13:45 | 是否创建了特性分支 | 引导创建分支，不要在 develop 直接开发 |
| 14:00 | 是否完成了设计分析 | 催促完成，不要跳过直接写代码 |
| 14:15 | proposal + tasks 质量 | 快速审核，引导修改 |
| 14:40 | TDD 是否执行 | 检查是否先写测试 |
| 15:00 | 需求变更前检查进度 | 落后的先完成主需求核心部分 |
| 15:30 | 变更后 proposal 是否更新 | 提醒更新文档 |
| 16:15 | 是否执行了三方交叉审查 | 引导使用 Codex/Gemini |
| 16:25 | 是否提交到特性分支 | 引导 git commit + push |
