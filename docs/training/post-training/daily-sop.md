# 训后日常 SOP：AI 驱动开发标准操作流程

> 从培训结束后的第一个工作日开始执行

---

## 核心制度

### 制度1：MR 附带 OpenSpec（渐进式推广）

**分两阶段推广，避免形式主义反弹：**

#### 第一阶段：试点期（第 1-2 周）
- 仅针对**预估 > 1 人天的新功能或重要重构**，要求附带 OpenSpec
- Bug 修复、配置调整、单点优化**不要求** OpenSpec
- 选择 1-2 个核心业务线先行试点
- 收集反馈，优化流程

#### 第二阶段：全面推广（第 3 周起）
- 根据试点反馈调整标准后全面推广
- **从培训后第 3 周的周一起，符合条件的 MR 必须包含以下内容：**

```markdown
## OpenSpec 链接
- Proposal: openspec/changes/xxx/proposal.md
- Tasks: openspec/changes/xxx/tasks.md

## 变更类型
- [ ] 新功能（需要完整 OpenSpec）
- [ ] Bug 修复（免 OpenSpec，描述清楚即可）
- [ ] 配置调整（免 OpenSpec）
- [ ] 重构（需要 OpenSpec）
```

**豁免条件**：
- Bug 修复：仅需在 MR 描述中写清问题和修复方案
- 配置调整：仅需在 MR 描述中写清变更内容
- 紧急 Hotfix：先修复后补 OpenSpec（24 小时内）

### 制度2：每周 AI 提效案例分享

**每周五站会新增 15 分钟分享环节：**

- 每组轮流，每周 1 个组分享
- 分享内容：
  1. 本周用 Claude Code 解决了什么问题
  2. 哪个 Prompt 技巧最有效
  3. 踩了什么坑，怎么爬出来的
- 鼓励分享失败案例（比成功案例更有学习价值）

### 制度3：AI Champion 机制

**每组 1-2 名 Champion，职责详见 [champion-guide.md](champion-guide.md)**

## 日常开发流程（新 SOP）

### 小任务（Bug 修复、配置调整）

```
1. 在 Claude Code 中描述问题
2. 让 Claude Code 分析代码定位问题
3. TDD：先写测试复现 Bug → 修复代码 → 测试通过
4. 提交 MR（描述清楚问题和修复方案）
```

### 中任务（单模块新功能）

```
1. 创建 OpenSpec proposal（借助 Claude Code）
2. 编写 tasks.md（bite-sized 步骤）
3. 同事/Leader 审核 proposal
4. 按 tasks.md 逐步实现（TDD）
5. 用 Codex/Gemini 交叉审查
6. 提交 MR（附带 OpenSpec 链接）
7. 归档 OpenSpec
```

### 大任务（跨模块/架构变更）

```
1. 创建 OpenSpec proposal + 设计文档
2. 团队评审 proposal
3. 细化实现计划（writing-plans）
4. 分步实现（可用 subagent 并行）
5. 阶段性 Code Review
6. 完成后归档 + specs/ 同步
```

## Prompt 最佳实践（团队共享）

### 好的 Prompt 模板

**需求分析**：
```
我需要实现 [功能描述]。
背景：[为什么需要这个功能]
约束：[技术约束、性能要求等]
请先分析需求，然后创建 OpenSpec proposal。
```

**Bug 修复**：
```
问题现象：[具体描述]
错误日志：[贴日志]
复现步骤：[1. 2. 3.]
我怀疑是 [你的判断]，请分析并修复。
```

**代码审查**：
```
请审查 [文件路径] 的实现：
1. 是否有安全隐患？
2. 性能是否有优化空间？
3. 错误处理是否完善？
4. 是否符合团队规范？
```

### 坏的 Prompt（避免）

```
❌ "帮我写个登录功能"      → 太模糊
❌ "修一下这个 bug"         → 没有上下文
❌ "优化一下这段代码"       → 没有优化方向
❌ "重构这个模块"           → 没有目标
```

## 工具使用规范

### Claude Code 使用场景

| 场景 | 推荐做法 |
|------|---------|
| 理解陌生代码 | 让 Claude 读代码并解释 |
| 写新功能 | 先写 OpenSpec，再让 Claude 实现 |
| 修 Bug | 给上下文（日志、复现步骤），让 Claude 分析 |
| Code Review | 让 Claude 审查，关注安全和性能 |
| 写测试 | 描述场景，让 Claude 生成测试代码 |

### MCP 工具使用场景

| 工具 | 场景 |
|------|------|
| Codex | 后端代码审查、算法设计审查 |
| Gemini | 前端组件实现、大规模代码分析、文档生成 |

| 指标 | 说明 |
|------|------|
| OpenSpec 覆盖率 | 附带 OpenSpec 的 MR 占比 |

## 常见问题 FAQ

**Q: 小改动也要写 OpenSpec 吗？**
A: 不用。Bug 修复、配置调整等小改动可以跳过。判断标准：预估工作量 > 1 人天的新功能或重要重构才需要 OpenSpec。

**Q: OpenSpec 写不好怎么办？**
A: 让 Claude Code 帮你写！描述需求后让它生成 proposal，然后你审核调整。

**Q: AI 生成的代码质量不行怎么办？**
A: 优化你的 Prompt。给更多上下文、约束和示例。记住"garbage in, garbage out"。

**Q: 遇到 AI 解决不了的问题怎么办？**
A: 正常！AI 是工具不是万能的。这时候人类介入，用传统方式解决，然后分享经验。

**Q: 我不想用 AI 怎么办？**
A: OpenSpec 流程是必须的（它不依赖 AI）。Claude Code 是推荐的，给自己一个月试用期。
