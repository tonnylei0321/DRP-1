## 破冰第一步

拿到代码后按以下顺序操作：

1. `cd scaffold/react/ && npm install` — 安装依赖
2. `npm run dev` — 启动开发服务器，浏览器打开确认页面可访问
3. `npm test` — 运行测试（当前无测试，确认命令可用）
4. **从 develop 创建自己的特性分支**：
   ```bash
   git checkout develop
   git checkout -b feature/training-{你的名字}
   ```
5. 启动 Claude Code：`claude`
6. 发送第一条指令：`请阅读 scaffold/react/src/ 目录下的代码结构和类型定义，我需要实现一个多步表单向导`

## 实践流程（严格按此顺序执行）

> 对齐 CLAUDE.md 规范：设计先行 → 三方审核 → 提案 → TDD 实现 → 三方审查

### 阶段1：需求分析与设计（13:40-14:00）

1. 在 Claude Code 中粘贴下方"主需求"，让 Claude 进行需求分析
2. 让 Claude 分析组件结构设计、状态管理方案（Hook 设计）
3. **三方设计审核**：
   - 让 Claude 调用 Codex 审查组件拆分合理性和状态管理方案
   - 让 Claude 调用 Gemini 审查用户交互场景覆盖度
   - 三方达成一致后，确认设计方案

### 阶段2：编写 OpenSpec 提案（14:00-14:20）

1. 用 Claude Code 生成 OpenSpec 提案：
   - `proposal.md`：Why + What Changes + Impact
   - `tasks.md`：bite-sized 实现步骤（每步含文件路径、代码要点、验证命令）
2. **三方提案审核**：Codex 审查组件接口设计，Gemini 审查场景覆盖
3. 讲师审核通过 → **关卡1打卡**

### 阶段3：TDD 实现（14:20-15:00）

1. 按 tasks.md 步骤顺序实现
2. **每个 task 严格 TDD**：先写测试（RED）→ 实现代码（GREEN）→ 重构
3. **关卡2打卡**：三步表单可正常流转 + 测试通过

### 阶段4：需求变更（15:00-16:00）

**15:00-15:15 更新文档（强制停顿，禁止写代码）**：
1. 讲师宣布变更需求（见下方"中途变更"）
2. 更新 proposal.md：在 What Changes 中新增变更内容
3. 更新 tasks.md：新增实现步骤
4. **三方变更审核**：Codex/Gemini 审核变更方案的合理性

**15:15-16:00 变更实现**：
1. 按更新后的 tasks.md 继续 TDD 实现
2. 处理边界条件
3. 补充测试用例

### 阶段5：三方代码审查与验证（16:00-16:30）

1. **Codex 审查**：状态管理是否有内存泄漏、localStorage 异常处理
2. **Gemini 审查**：组件是否有不必要重渲染、无障碍访问（a11y）
3. 根据审查意见修复问题
4. 运行全部测试，确认通过
5. 提交到特性分支：`git add . && git commit -m "feat: 多步表单向导"`
6. **关卡3打卡**：三方审查通过 + 测试全绿

## 业务背景

公司内部管理系统需要一个员工入职信息收集表单。表单内容较多，需要分步填写，避免用户一次面对太多字段。当前只有一个简单的单页表单，用户体验差，经常漏填。

## 需求描述

### 主需求（14:00-15:00 完成）

实现三步表单向导：

```
步骤1：基本信息          步骤2：工作信息          步骤3：确认提交
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 姓名 *        │    │ 部门（下拉）* │    │ [信息汇总展示] │
│ 手机号 *      │ →  │ 职位 *        │ →  │              │
│ 邮箱 *        │    │ 入职日期 *    │    │ [提交] [返回] │
│ [下一步]      │    │ [上一步][下一步]│   │              │
└──────────────┘    └──────────────┘    └──────────────┘
```

**具体要求**：
1. 步骤指示器（显示当前在第几步）
2. 每步表单验证：
   - 姓名：必填，2-20 字符
   - 手机号：必填，11 位数字
   - 邮箱：必填，合法邮箱格式
   - 部门：必填，从预定义列表选择
   - 职位：必填
   - 入职日期：必填，不早于今天
3. 验证不通过不能进入下一步
4. 第三步展示所有填写信息的汇总
5. 提交按钮点击后 console.log 提交数据（模拟 API 调用）

### 中途变更（15:00 讲师宣布）

新增"步骤回退 + 数据保持"功能：

**变更要求**：
1. 点击上一步时，之前填写的数据必须保留
2. 新增浏览器刷新后数据恢复功能（localStorage 持久化）
3. 新增"重置表单"按钮，清空所有数据（含 localStorage）
4. 步骤指示器支持点击跳转（但只能跳到已填写过的步骤）

### 边界条件
- 直接访问 URL 应从第一步开始
- 手机号输入非数字时实时提示
- 部门列表：技术部、产品部、设计部、运营部、人事部

### 交叉审查（三方分工）
```
# 第一步：Codex 审查代码质量和安全性
请 Codex 审查表单向导的实现：
1. 表单状态是否有内存泄漏风险？
2. localStorage 操作是否有异常处理？
3. useFormWizard Hook 是否有不必要的重渲染？
4. 输入数据是否有 XSS 风险？

# 第二步：Gemini 审查功能覆盖度和 UI 合理性
请 Gemini 审查表单向导的实现：
1. 三步表单的用户交互流程是否顺畅？
2. 无障碍访问（a11y）是否合格？
3. 表单验证的场景是否覆盖完整？
4. 组件拆分和 Props 设计是否合理？

# 第三步：验证实现是否偏离设计
对照 proposal.md 检查：
1. 实现的功能是否和 What Changes 一致？
2. 是否有未在 proposal 中声明的变更？
```
