# 学员实战流程引导

> 本文件是你下午实战的操作指南，按顺序执行即可。

---

## 流程总览

```
破冰准备（5min）
    ↓
需求分析与设计（10min）
    ↓
编写 OpenSpec 提案（5min）→ 关卡1打卡
    ↓
TDD 实现主需求（45min）→ 关卡2打卡
    ↓
需求变更：更新文档（10min）+ 实现变更（35min）
    ↓
三方代码审查 + 验证（15min）→ 关卡3打卡
```

---

## 阶段1：破冰准备（5min）

1. 进入你的技术栈对应的脚手架目录
2. 安装依赖、运行测试，确认环境正常
3. **从 develop 创建自己的特性分支**：
   ```bash
   git checkout develop
   git checkout -b feature/training-{你的名字}
   ```
4. 启动 Claude Code：`claude`
5. 发送第一条指令，让 Claude 阅读脚手架代码结构

---

## 阶段2：需求分析与设计（10min）

1. 在 Claude Code 中粘贴题目的"主需求"部分
2. 让 Claude 进行需求分析，给出组件/模块设计方案
3. **三方设计审核**：
   - 让 Claude 调用 Codex 审查技术可行性
   - 让 Claude 调用 Gemini 审查场景覆盖度
   - 三方达成一致后，确认设计方案

> 这一步禁止写业务代码，只做分析和设计。

---

## 阶段3：编写 OpenSpec 提案（5min）

1. 用 Claude Code 生成 OpenSpec 提案：
   - `proposal.md`：Why（为什么做）+ What Changes（做什么）+ Impact（影响范围）
   - `tasks.md`：bite-sized 实现步骤（每步含文件路径、代码要点、验证命令）
2. **三方提案审核**：Codex 审查 API 设计，Gemini 审查场景覆盖
3. 讲师审核通过

**→ 关卡1打卡：proposal.md + tasks.md 完成，群里发截图**

---

## 阶段4：TDD 实现主需求（45min）

1. 按 tasks.md 步骤顺序实现
2. **每个 task 严格 TDD**：
   - 先写测试（RED）→ 运行测试，确认失败
   - 实现代码（GREEN）→ 运行测试，确认通过
   - 重构（REFACTOR）→ 优化代码，测试仍通过
3. 遇到问题先问 Claude Code，再问讲师

**→ 关卡2打卡：主功能测试通过，群里发截图**

---

## 阶段5：需求变更（45min）

### 5a. 更新文档（10min）— 强制停顿，禁止写代码

1. 讲师宣布变更需求
2. 更新 proposal.md：在 What Changes 中新增变更内容
3. 更新 tasks.md：新增实现步骤
4. **三方变更审核**：Codex/Gemini 审核变更方案的合理性

### 5b. 实现变更（35min）

1. 按更新后的 tasks.md 继续 TDD 实现
2. 处理边界条件
3. 补充测试用例

---

## 阶段6：三方代码审查 + 验证（15min）

### 6a. 三方代码审查（10min）

分两步调用，不要合并给一个 AI：

```
# 第一步：Codex 审查代码质量和安全性
请 Codex 审查我的实现：
1. 代码质量和安全隐患
2. 边界条件处理
3. 性能问题

# 第二步：Gemini 审查功能覆盖度
请 Gemini 审查我的实现：
1. 功能是否覆盖完整
2. 实现是否偏离 proposal 需求
3. 测试用例是否充分
```

根据审查意见修复问题。

### 6b. 最终验证 + 提交（5min）

1. 运行全部测试，确认通过
2. 检查 tasks.md 中所有任务已完成
3. 提交到特性分支：
   ```bash
   git add .
   git commit -m "feat: 完成训练任务 - {你的题目名}"
   ```

**→ 关卡3打卡：三方审查通过 + 测试全绿，群里发截图**

---

## 核心原则（牢记）

1. **先设计后编码** — 不写 proposal 不动手
2. **先测试后实现** — 不写测试不写代码（TDD）
3. **先更新文档后变更** — 需求变了先改 proposal
4. **你是主人，AI 是顾问** — 审查 AI 生成的每一行代码
5. **三方分工明确** — Codex 查质量安全，Gemini 查功能覆盖，不要混用
