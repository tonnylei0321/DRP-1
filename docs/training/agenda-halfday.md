## 第二部分：分轨实战（09:55 - 12:00）

> **核心流程**：设计先行 → 三方审核 → 提案 → TDD 实现 → 需求变更 → 三方代码审查

### 09:55 - 10:15 | 破冰 + 需求设计 + OpenSpec 提案（20min）

**流程**：
1. **09:55-10:00**（5min）— 题目讲解 + 破冰
   - 讲师讲解题目
   - 学员安装依赖、运行测试、启动 Claude Code
   - **每人从 develop 创建自己的特性分支**

2. **10:00-10:10**（10min）— 需求分析 + 设计 + 三方审核
   - 在 Claude Code 中进行需求分析和架构设计
   - 三方设计审核：Codex 审查技术可行性，Gemini 审查场景覆盖度

3. **10:10-10:15**（5min）— 编写 OpenSpec 提案
   - 用 Claude Code 生成 proposal.md + tasks.md
   - 三方提案审核
   - **关卡1打卡**：proposal.md + tasks.md 完成，群里发截图

---

### 10:15 - 11:00 | TDD 实现主需求（45min）

**流程**：
1. 按 tasks.md 步骤顺序实现
2. **严格 TDD**：先写测试（RED）→ 实现代码（GREEN）→ 重构
3. 讲师巡场处理卡壳问题
4. **关卡2打卡**：主功能测试通过，群里发截图

---

### 11:00 - 11:45 | 需求变更（45min）

**流程**：
1. **11:00-11:10**（10min）— 变更投放 + 更新文档（强制停顿，禁止写代码）
   - 讲师宣布需求变更
   - 更新 proposal.md 和 tasks.md
   - 三方变更审核

2. **11:10-11:45**（35min）— 变更实现
   - 按更新后的 tasks.md 继续 TDD 实现
   - 处理边界条件，补充测试

---

### 11:45 - 12:00 | 三方代码审查 + 验证（15min）

**流程**：
1. **11:45-11:55**（10min）— 三方代码审查
   - Codex 审查代码质量和安全性
   - Gemini 审查功能覆盖度和实现一致性
   - 根据审查意见修复问题

2. **11:55-12:00**（5min）— 最终验证 + 提交
   - 运行全部测试，确认通过
   - 提交到特性分支
   - **关卡3打卡**：三方审查通过 + 测试全绿，群里发截图

**讲师巡场检查点**：
| 时间 | 检查内容 | 发现问题怎么办 |
|------|---------|--------------|
| 10:00 | 是否创建了特性分支 | 引导创建分支 |
| 10:10 | 是否完成了设计分析 | 催促完成 |
| 10:30 | TDD 是否执行 | 检查是否先写测试 |
| 11:00 | 主需求进度 | 落后的先完成核心部分 |
| 11:10 | 变更后文档是否更新 | 提醒更新 |
| 11:50 | 三方审查是否执行 | 引导使用 Codex/Gemini |
| 11:55 | 是否提交到特性分支 | 引导 git commit |
